# Implementation Plan — Video Chat Room

- [ ] **1. Настройка проекта и инфраструктуры**
  - Инициализация Node.js проекта (`npm init`).
  - Установка зависимостей: `express`, `socket.io`, `cors`, `uuid` (для генерации ID).
  - Настройка React-приложения (например, Create React App или Vite).
  - Настройка структуры папок: `/server`, `/client/src`.
  - Настройка скриптов `package.json` для одновременного запуска клиента и сервера (concurrently).
  - _Requirements: F-01..F-04 (общие)_

- [ ] **2. Реализация серверной логики (Node.js + Socket.io)**
  - **2.1** Создание HTTP-сервера с Express и подключение Socket.io.
  - **2.2** Разработка `RoomManager` – объекта для хранения комнат в памяти с методами: `createRoom`, `addParticipant`, `removeParticipant`, `getRoom`, `deleteRoomIfEmpty`.
  - **2.3** Обработка события `connection`: генерация socket.id.
  - **2.4** Обработка `join-room`: валидация имени, проверка лимита (атомарно), добавление участника, отправка `room-state` новичку, широковещание `participant-joined` остальным.
  - **2.5** Обработка `leave-room` и `disconnect`: удаление участника, отправка `participant-left`, удаление комнаты при опустении.
  - **2.6** Обработка `signal`: пересылка сигнальных сообщений между участниками (проверка, что оба в комнате).
  - **2.7** Обработка `chat-message`: добавление сообщения в историю, широковещание `chat-message`.
  - **2.8** Добавление логирования основных событий для отладки.
  - _Requirements: F-02, F-05, F-17, F-18, F-12, F-13, F-14, F-15_

- [ ] **3. Разработка клиентского приложения (React) – общая структура**
  - **3.1** Создание маршрутизации (React Router): `/` – стартовый экран, `/:roomId` – комната.
  - **3.2** Создание хука `useSocket` для подключения к серверу и управления событиями (подписка, отправка).
  - **3.3** Создание хука `useWebRTC` для управления подключениями и медиа-потоками.
  - **3.4** Создание контекста (или store) для глобального состояния приложения (участники, сообщения, медиа-статус).
  - _Requirements: общие_

- [ ] **4. Стартовый экран (StartScreen)**
  - **4.1** Компонент с полем ввода имени и кнопкой «Создать комнату».
  - **4.2** Валидация имени: не пустое, не только пробелы, длина ≤30, допускаются только безопасные символы (регулярка).
  - **4.3** Генерация `roomId` (например, `uuidv4()`) и переход на `/roomId` (или если уже есть roomId из URL – использовать его).
  - **4.4** При открытии страницы с `roomId` – сразу показать StartScreen с уже заполненным полем (если есть параметр), и кнопка «Войти».
  - _Requirements: US-1, F-01, F-02_

- [ ] **5. Экран комнаты – основные компоненты**
  - **5.1** Создание компонента `Room` – обёртка, подключается к Socket, получает `room-state`.
  - **5.2** При получении `room-state` инициализирует WebRTC-подключения к существующим участникам.
  - **5.3** Отображение видеосетки (`VideoGrid`) и панели управления (`Controls`).
  - **5.4** Отображение чата (`ChatPanel`) и списка участников (можно внутри чата или отдельно).
  - _Requirements: F-06, F-07, F-16_

- [ ] **6. Видеосетка и плитки**
  - **6.1** Компонент `VideoGrid` – адаптивная сетка (1–4 плитки), использует CSS Grid или Flexbox.
  - **6.2** Компонент `VideoTile` – отображение видео (HTMLVideoElement), overlay с именем.
  - **6.3** Добавление self-view (локальное видео) – отдельная плитка с пометкой «Я» (или ваше имя).
  - **6.4** Отображение статуса микрофона (иконка перечёркнутого микрофона, если выключен).
  - **6.5** Отображение заглушки при отсутствии видео – силуэт человека (SVG или иконка) и имя.
  - **6.6** Реализация динамического добавления/удаления плиток при входе/выходе участников (через обновление состояния).
  - _Requirements: F-07, F-08, F-09, F-10, US-7, US-12_

- [ ] **7. WebRTC-логика (управление соединениями)**
  - **7.1** Создание `RTCPeerConnection` для каждого удалённого участника при его появлении (или при своём входе).
  - **7.2** Получение локального медиа-потока (`getUserMedia`) с камерой и микрофоном.
  - **7.3** Добавление треков локального потока в каждое соединение (offer).
  - **7.4** Обработка входящих офферов, ответов, ICE-кандидатов через событие `signal`.
  - **7.5** Установка удалённых потоков на `video`-элементы (добавление track).
  - **7.6** Обработка ошибок `getUserMedia` (отказ, отсутствие устройств) – показ сообщений, но продолжение работы.
  - **7.7** Обработка изменения состояния соединения (onconnectionstatechange) – если соединение упало, уведомлять пользователя? (по PRD – просто удаляем участника при дисконнекте).
  - _Requirements: F-06, F-08, F-18, US-6, US-11_

- [ ] **8. Управление медиа (включение/выключение камеры, микрофона)**
  - **8.1** Кнопки включения/выключения микрофона: переключение дорожки (track.enabled = false/true), обновление иконки.
  - **8.2** Кнопки включения/выключения камеры: остановка/запуск видеодорожки (при выключении – освободить устройство, при включении – запросить заново). Использовать `getUserMedia` повторно для видео, или использовать `track.stop()` и потом `replaceTrack`? Рекомендуется при выключении камеры остановить трек и удалить его из всех соединений (или просто отключить enabled). Но для освобождения устройства нужно остановить трек. Лучше: при выключении – `track.stop()` и удалить из всех peer connections; при включении – запросить новый поток и добавить треки. Это сложнее, но соответствует требованию «физически освободить камеру».
  - **8.3** Отображение состояния на своей плитке (иконки).
  - **8.4** Система должна синхронизировать состояние? По PRD – нет, состояние локальное; остальные видят только индикаторы через локальный UI (это не передаётся по сигнальному каналу, только через медиа). Но можно передавать состояние через чат или сигнал, но не обязательно.
  - _Requirements: F-09, F-10, US-7_

- [ ] **9. Текстовый чат**
  - **9.1** Компонент `ChatPanel` – список сообщений с авто-прокруткой вниз при новых сообщениях.
  - **9.2** Поле ввода и кнопка отправки (или Enter).
  - **9.3** Валидация: не пустое, не только пробелы.
  - **9.4** При получении `chat-message` от сервера – добавлять в состояние и рендерить (с именем отправителя и временем HH:MM).
  - **9.5** При получении `participant-joined` и `participant-left` – добавлять системные сообщения в чат (согласно PRD).
  - **9.6** Экранирование текста сообщений и имён при рендеринге (использовать `dangerouslySetInnerHTML` нельзя – использовать простой текст).
  - _Requirements: F-12, F-13, F-14, F-15, US-8, US-9_

- [ ] **10. Копирование ссылки и приглашение**
  - **10.1** Кнопка «Скопировать ссылку» – копирует текущий URL в буфер обмена (использовать `navigator.clipboard.writeText`).
  - **10.2** Показать уведомление об успешном копировании (toast или небольшой всплывающий текст).
  - _Requirements: F-03, US-3_

- [ ] **11. Обработка крайних случаев и ошибок (UI)**
  - **11.1** Отображение сообщения «Комната заполнена» при получении события `room-full` – с кнопкой «Повторить».
  - **11.2** Отображение сообщения об ошибке доступа к устройствам (`NotAllowedError`, `NotFoundError`) – не блокируя вход.
  - **11.3** Отображение сообщения, если сервер недоступен (событие `connect_error`).
  - **11.4** Проверка поддержки WebRTC – если нет, показываем сообщение и не даём войти.
  - **11.5** Обработка авто-прокрутки чата – прокручивать вниз при каждом новом сообщении.
  - **11.6** Обработка политики autoplay – при первом клике на кнопку «Войти» или «Создать комнату» разрешить аудио-контекст (можно создать аудио-контекст и сразу его воспроизвести/возобновить).
  - _Requirements: US-5, US-11, US-12, US-13_

- [ ] **12. Выход и завершение сессии**
  - **12.1** Кнопка «Выйти» – отправляет `leave-room`, закрывает все peer-соединения, останавливает медиа, переходит на стартовый экран.
  - **12.2** При закрытии вкладки – обрабатываем `beforeunload`? По PRD – это приравнивается к выходу, но сервер сам обнаружит `disconnect`. Можно дополнительно отправить `leave-room` при `beforeunload`, но это не гарантируется. Достаточно полагаться на `disconnect`.
  - **12.3** При обрыве соединения участника – сервер удаляет его, остальные получают `participant-left`; клиентское состояние обновляется автоматически.
  - _Requirements: F-17, F-18_

- [ ] **13. Стилизация и адаптивность**
  - **13.1** Базовый CSS (или модули) для всех компонентов – читаемый, минималистичный.
  - **13.2** Адаптивная видеосетка: при 1 участнике – одна плитка по центру, при 2 – две колонки, при 3 – три или 2+1, при 4 – сетка 2×2.
  - **13.3** Минимальная ширина экрана 1024px – десктоп.
  - **13.4** Цветовая схема – нейтральная, кнопки с иконками.
  - **13.5** Оверлей имени на плитках.
  - _Requirements: F-07, дизайн_

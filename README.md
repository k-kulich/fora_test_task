# Video Chat Room — тестовое задание

## Описание

Веб-приложение для группового видеозвонка со встроенным текстовым чатом. Пользователь вводит имя, создаёт или присоединяется к комнате по ссылке, после чего общается в реальном времени — видео, аудио и текстовые сообщения.

Приложение реализовано в рамках тестового задания для Fora Soft на основе [PRD](prd-video-chat-room.md) и [TDD](design-video-chat-room.md). Основной упор сделан на скорость разработки и соответствие функциональным требованиям.

## Стек технологий

| Компонент | Технология | Назначение |
|-----------|------------|------------|
| **Frontend** | React (Vite) + LiveKit Client SDK | Интерфейс, захват медиа, подключение к SFU |
| **Backend** | Node.js + Express + LiveKit Server SDK | Генерация токенов, управление комнатами (in-memory) |
| **SFU (медиасервер)** | [LiveKit](https://livekit.io/) (self-hosted) | Маршрутизация видео/аудио, сигнализация, ICE/STUN |
| **TURN/STUN** | публичные STUN-серверы Google (опционально coturn) | Обход NAT для установки соединения |
| **Развёртывание** | Docker Compose | Запуск всех сервисов одной командой |

## Почему LiveKit, а не «чистый» WebRTC + Socket.io?

В исходном PRD (тест-задание) был заявлен стек: **JavaScript (ES6+), Node.js, React, Socket.io, WebRTC** с топологией mesh (P2P). Однако **файлы-ссылки на дизайн-документы и планы задач** (`prd-design.mdc`, `prd-tasks.mdc`) долгое время не открывались, и доступ к ним удалось получить только за день до дедлайна.

В такой ситуации **выбор LiveKit** был продиктован следующими причинами:

- **Готовый SFU** — LiveKit уже содержит всю сигнализацию, маршрутизацию медиа, ICE-обработку и управление участниками. Это позволило реализовать работоспособный MVP за **несколько часов**, а не за дни.
- **Меньше кода** — не пришлось писать собственный сигнальный сервер на Socket.io, обрабатывать SDP-офферы/ответы и ICE-кандидаты, а также управлять mesh-соединениями (6 соединений на 4 участников).
- **Масштабируемость** — при необходимости можно легко увеличить лимит участников без переписывания архитектуры.
- **Стабильность** — LiveKit уже проверен в боевых условиях, что снижает риск ошибок при подключении.
- **Быстрая интеграция** — клиентский SDK для React и серверный SDK для Node.js позволяют сосредоточиться на бизнес-логике, а не на низкоуровневом WebRTC.

Таким образом, функционально приложение полностью соответствует требованиям PRD (видео-звонок до 4 участников, чат, управление камерой/микрофоном, системные события), но **медиаслой реализован через LiveKit SFU**, а не через mesh-соединения.

При этом __доступ к файлам__ я смогла получить только __вчера__, а потому __версия WebRTC лежит в ветке web-rtc__ этого же репозитория. Однако более стабильной и полной является версия на LiveKit. Запуск WebRTC версии возможен через прямой запуск `npm run dev` в терминале для фронтенда и бэкенда отдельно.

## Установка и запуск (локальная разработка)

### Требования
- Node.js
- Docker и Docker Compose (для LiveKit)
- Git

### 1. Клонирование репозитория
```bash
git clone <url>
```

### 2. Настройка окружения
Создайте файл `.env` в корне проекта (или скопируйте `.env.example`):
```bash
LIVEKIT_API_KEY=devkey
LIVEKIT_API_SECRET=secret
PUBLIC_IP=127.0.0.1
PUBLIC_URL=http://localhost:3000
```

Для локальной разработки можно использовать ключи по умолчанию (`devkey`/`secret`). Для продакшена следует сгенерировать свои ключи.

### 3. Запуск LiveKit (через Docker)
```bash
docker-compose up -d livekit
```

LiveKit будет доступен по адресу `ws://localhost:7880`.

### 4. Запуск бэкенда
```bash
cd backend
npm install
npm start
```
Бэкенд запустится на порту 3000.

### 5. Запуск фронтенда
```bash
cd frontend
npm install
npm run dev
```
Фронтенд будет доступен по адресу `http://localhost:5173`.

## Функциональные возможности

- **Создание и присоединение к комнатам** — по имени или случайному идентификатору.
- **Видео/аудио связь** до 4 участников (через LiveKit SFU).
- **Текстовый чат** в реальном времени (через DataChannel LiveKit).
- **Управление камерой и микрофоном** (включить/выключить) с физическим освобождением устройств.
- **Демонстрация экрана** (только один участник за раз).
- **Системные уведомления** о входе/выходе участников.
- **Отображение имён** и индикация статуса (микрофон, камера).
- **Автоматическое удаление комнаты** после выхода последнего участника.

## Примечания

- Проект разрабатывался с использованием **AI-ассистента** (DeepSeek по определенным причинам) в соответствии с подходом, описанным в тестовом задании (PRD → TDD → план задач → реализация).
- В связи с задержкой доступа к файлам-шаблонам (`prd-design.mdc`, `prd-tasks.mdc`), реализация выполнена на **LiveKit** вместо «чистого» WebRTC + Socket.io. Это решение позволило уложиться в сроки и получить стабильный продукт.
- Для локального тестирования не требуется TURN-сервер — достаточно STUN-серверов Google.
- Для продакшена рекомендуется добавить собственный TURN-сервер (например, coturn) для гарантированной связности за строгими NAT.
